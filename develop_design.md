# 开发设计文档

本文档聚焦“做什么、为什么做、怎么拆分模块与协作”，并尽量避免与契约文档重复。

---

## 1. 项目背景与目标

### 1.1 项目背景

本项目是一个视频分析助手：用户提供外部平台视频链接或本地视频文件后，后端对视频进行下载/解析，调用 AI 将视频内容划分为章节（作为后续分析与产物的唯一准则），并并发执行两个子分析流程生成可渲染产物，最终返回给前端展示与编辑。

### 1.2 目标用户与使用方式

产品提供两种主要使用方式：

1. Web/Docker：开发者或高级用户通过源码编译，或拉取 Docker 镜像在浏览器使用。
2. 桌面版：普通用户下载桌面应用直接使用；桌面端负责启动/管理后端服务并提供本地能力。

### 1.3 MVP 功能范围

MVP 目标是闭环完成：导入视频 → 分析 → 查看 → 编辑 → 持久化 → 下次可再次打开。

包含：

- 支持链接导入与本地视频导入。
- 后端异步任务（Job）执行分析流程，前端可查看进度并在失败时给出可理解的错误。
- 以“章节切片结果”为唯一准则：思维导图与重点摘要均基于同一章节划分。
- 并发两个子 agent 生成产物：
  - 子 agent A：思维导图（nodes/edges）。
  - 子 agent B：章节重点与关键帧（每章重点文本 + 关键帧截图 + 起止时段）。
- 多项目持久化：可保存多个视频分析结果，并可列出/打开。
- 前端可编辑：用户可对生成的笔记与思维导图进行手动修改并保存。

非目标（MVP 不承诺）：

- 账号体系/云同步/多端同步。
- 大规模分布式任务调度。
- 完整 RAG、Notion/PDF 等所有导出形态（可以作为后续扩展）。

### 1.4 术语与约束

- Project：一个视频笔记项目，包含视频来源、多个分析 Job、最新 Result 指针等。
- Job：一次长耗时分析任务，具有状态机与进度。
- Result：一次分析完成的可渲染产物（思维导图、笔记、章节、关键帧等）。
- Chapter：由“切片流程”输出的章节，后续所有分析与 UI 跳转以 Chapter 为准。

关键约束：

- 本地优先：默认所有数据存于本机，确保可离线浏览历史结果。
- 跨端一致：Web/Docker 与桌面端使用同一后端协议与数据契约。
- 可恢复：刷新页面/重启应用后仍能查询 Job 状态与读取历史 Result。

## 2. 技术选型（暂定）

### 2.1 核心技术栈

- 前端 (UI/UX)
  - Next.js (React)：视图与交互。
  - Tailwind CSS + Shadcn UI：UI 组件与主题。
  - TipTap：富文本编辑器（可扩展节点：时间戳、截图卡片等）。
  - React Flow：思维导图渲染与编辑。

- 桌面端容器
  - Tauri：桌面壳与后端 Sidecar 管理。

- 后端 (AI & 处理引擎)
  - FastAPI (Python)：API 服务与任务编排。
  - yt-dlp：视频/音频下载（外部平台）。
  - FFmpeg：音频提取、抽帧、时间点截图。

- 存储层
  - SQLite + sqlite-vec：项目/任务/产物元数据与向量检索扩展。
  - 本地文件系统：视频缓存（可选）、截图、导出资源等。

## 3. 总体架构与关键流程

### 3.1 架构形态

采用 Sidecar 架构：桌面端（Tauri）作为主进程，负责启动并管理后端服务；Web/Docker 形态直接运行后端服务并由浏览器访问。

### 3.2 端到端流程（概览）

1. 导入
   - URL：后端使用 yt-dlp 获取可处理的视频/音频。
   - 本地文件：后端直接读取（桌面端为主）。

2. 预处理
   - 统一生成音频流/音频文件与基本元信息（时长、格式）。

3. 切片（唯一准则）
   - 使用 AI/规则对转录内容进行章节划分，输出 Chapter 列表：每章 start_ms/end_ms/title/summary。
   - 后续分析与 UI 展示必须使用这一份 Chapter 作为唯一时间与结构依据。

4. 并发子分析（两个子 agent 并行）
   - 子 agent A（Mindmap Agent）：为每章及全局生成思维导图结构（nodes/edges）。
   - 子 agent B（Highlights+Keyframes Agent）：为每章提炼重点，并从视频中截取关键帧；同时给出重点内容对应时段。

5. 汇总产物
   - 将章节、思维导图、笔记/摘要、关键帧资源等组合为 Result 并落库。

6. 前端渲染与编辑
   - 渲染思维导图、章节列表、富文本笔记与关键帧。
   - 允许用户编辑笔记/导图并保存为新的最新状态（或生成新的 Result 版本）。

### 3.3 Job 状态与阶段建议

Job 作为长任务一等公民，建议以阶段（stage）表达当前步骤，以 progress 表达整体进度。

参考阶段（可按实现调整）：

- ingesting：导入/下载
- extracting_audio：音频提取
- transcribing：转写
- segmenting：章节切片（唯一准则）
- analyzing_mindmap：子 agent A
- analyzing_highlights：子 agent B
- extracting_keyframes：抽帧（可与子 agent B 合并/穿插）
- assembling_result：汇总落库

原则：前端只展示后端提供的 status/progress/stage，不自行推断流程。

### 3.4 产物与一致性要求

为保证“一个章节定义统领所有视图”，约束如下：

- Chapters 由切片流程一次性产出并持久化；思维导图节点与重点列表必须能回溯到 chapter_id。
- 关键帧资源必须能映射到 project 与 chapter（或更细 time_ms）。
- Result 应可独立渲染：仅凭 Result 数据即可恢复笔记与导图的编辑态。

## 4. 契约与接口（单一真相）

### 4.1 契约位置

与“数据目录、命名、API、Job/Result 模型、SQLite 表结构”相关的契约定义，以 [api.md](api.md) 为唯一真相。

本文件仅补充落地注意事项与跨端约束，避免两处内容漂移。

### 4.2 落地注意事项

数据目录与路径：

- 后端建议只存相对 DATA_DIR 的相对路径；对外提供资源访问时，必须做路径安全限制，防止目录穿越。
- 导出类功能（若后续实现）应将图片引用统一写为导出目录内的相对路径，避免编辑态引用运行时目录。

进度推送：

- 事件流（SSE）为默认方案，前端断线/不支持时降级为轮询；心跳事件可避免误判断线。

鉴权与密钥：

- Web 端可将第三方 Key 放在请求头传递，后端不落盘，仅在内存使用。
- 桌面端优先使用系统安全存储（Keychain/凭据管理器）保存 Key 引用。

## 5. 工程结构设计（建议）

推荐采用 Monorepo 结构，便于共享类型、统一脚本与一键启动。

示例结构：

```
/video-note-pro
├── /apps
│   ├── /web              # Next.js 前端
│   └── /desktop          # Tauri 桌面端
│       └── src-tauri
├── /services
│   ├── /core             # FastAPI 后端
│   │   ├── /agents       # 切片与子 agent
│   │   ├── /processor    # yt-dlp/ffmpeg 等处理
│   │   ├── /db           # SQLite 持久化
│   │   └── main.py
│   └── /models           # 本地模型缓存（按需）
├── /shared               # 共享协议与类型
├── /data                 # 持久化数据根目录（开发默认）
└── docker-compose.yml
```

模块边界建议：

- services/core/processor：与媒体处理相关的纯函数/命令调用层。
- services/core/agents：与 LLM/AI 推理相关的策略层。
- services/core/jobs：Job 状态机、调度、取消、进度事件。
- services/core/db：项目/任务/结果/资源的落库与查询。

## 6. 里程碑与迭代计划

### 6.1 M0：骨架可运行

- 后端：health、projects、jobs 的最小可用接口；SQLite 初始化；本地文件目录创建。
- 前端：能创建项目、提交 Job、查看 Job 状态（轮询或 SSE）。

### 6.2 M1：分析闭环

- 完整流水线：导入 → 转写 → 切片 → 两子 agent 并发 → 汇总 Result。
- 产物可渲染：章节列表 + 思维导图 + 重点摘要 + 关键帧。

### 6.3 M2：编辑与持久化体验

- 前端编辑：TipTap 编辑保存；React Flow 编辑保存。
- 多项目：项目列表/删除/打开最新结果。

### 6.4 M3：桌面交付

- Tauri Sidecar：后端随桌面端启动；端口/进程管理；打包发布。

## 7. 风险与待决策

- 模型与性能：转写与切片的时延可能较高；需要可配置的模型与并发策略。
- 跨平台路径：Windows/macOS/Linux 路径与权限不同，必须使用统一的 DATA_DIR 约定并避免绝对路径泄露。
- 断线恢复：Job 事件流可能中断，必须保证轮询可恢复，并且 Result 可重放。
